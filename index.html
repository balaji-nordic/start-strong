Version Control (Save your future from the present you)

  • Git it from the get go!

    • Add tests from the very start

  • Bisectable commits​

    • Each commit should be self-contained and easy to isolate for debugging.

  • Branches

    • Long living branches are like long meetings.. no one likes them

  • Forks

    • Enforce forks if you don’t want git pull to feel like waiting for a pizza in a monorepo.
    • Keeps your main repo clean, but at the cost of a little freedom for the team. (You win some, you lose some.)

Code Reviews vs Pair Programming

  • Pair Programming
    • Two heads better than one - instant feedback 
    • No context switches 
  • Code Reviews
    • Structured, encourages independent evaluation
    • Context switches tend to slow the team down
  • Why not both?
    • Use a mix of pair programming for high-priority or complex areas and code reviews for the rest

  • Coverage tools
    • Gcov, lcov + genhtml
  • Coverage Goals
    • Statement Coverage: Shoot for 100 %
    • Branch Coverage: Aim for ~ 85%
    • Path Coverage: Depends on your project

Functional testing, Does the product work?

  • Check the basics

    • Either on your machine or via CI - because, lets be honest, sometimes you just don't want to run it yourself.

  When the nRF54L15 DK is selected

from behave import *
from nrfconnect_open import mytest
import time

boardConfigTest = mytest()

@when('nRF Connect for Desktop is opened')
def step_impl(context):
	boardConfigTest.opennRFConnectForDesktop()

@when(u'Board Configurator Open button is clicked')
def step_impl(context):
    boardConfigTest.openBoardConfigurator()

@then(u'the Board Configurator is opened')
def step_impl(context):
	assert boardConfigTest.checkBoardConfiguratorIsOpen()

@then(u'Close the Board Configurator')
def step_impl(context):
	boardConfigTest.closeActiveBoardConfigurator()

@then(u'Close nRF Connect for Desktop')
def step_impl(context):
	boardConfigTest.closenRFConnectForDesktop()

@given('the Board Configurator is opened')
def step_impl(context):
	boardConfigTest.opennRFConnectForDesktop()
	boardConfigTest.openBoardConfigurator()

@when('the nRF54L15 DK is selected')
def step_impl(context):
	boardConfigTest.selectnrf54l15DK()

@then('the nRF54L15 DK configuration is displayed')
def step_impl(context):
	assert boardConfigTest.checknRF54L15ConfigurationIsShown()

@then(u'select "{voltage:f}"v')
def step_impl(context, voltage):
	boardConfigTest.selectVoltage(voltage)

@then(u'Wait "{seconds:d}" seconds')
def step_impl(context, seconds):
	time.sleep(seconds)

Non-Functional Tests

  • Non-functional testing can help ensure your system is ready for the real world.
  • Is it fast?
    • Throughput tests
  • Is it reliable?
    • Long running tests!
  • Is it efficient?
    • If your system works but drains the battery in 10 mins, we've got a problem.
    • Power measurement tests
  • Is it tolerant to non-ideal conditions?
    • Heat
    • Humidity
    • Water splashing
    • Cold
  • Typically not run on Pull Requests or oncommit scope.

UX testing

  • ​“The User Will Never Do That”
    • Famous last words. Trust me, they will do that.
  • People Are Unpredictable
    • You’ll never guess what people actually do with your product until you watch them use it.
    • It’s both fascinating and terrifying.
  • But Wait, Can Embedded Systems Even Be UX Tested?
    • Absolutely! Embedded systems might not have flashy UIs, but they’re still used by humans. Their experience matters just as much as with web or desktop apps.
  • Find the Touchpoints
    • Identify where your system meets the user. Then design test cases for those interactions.
  • Test with Real People
    • Show your test cases to random subjects, ideally those who will use the system or interact with its users (hint: customer support folks know a lot about this).​

Debugging after deployment

Debugging after deployment - When things go Boom   💥

  • Design for Observability
    • Build your system so it tells you what’s going wrong, even after it’s out in the wild.
  • What if the system crashes?
    • Handle crashes like a pro: log faults to NVM (Non-Volatile Memory) or no-init RAM and read them out on restart. It’s like leaving breadcrumbs back to the problem.
  • Get creative with Remote monitoring
    • Find clever ways to monitor your software remotely - it'll save you a trip (and your sanity)
  • Asserts: Leave them on or off?
    • Minimalistic asserts can help, but whatever you choose, test the actual binary before shipping. You don’t want surprises after deployment!
  • Consider Memfault
    • A platform like Memfault allows you to remotely monitor, collect crash data, and more. Check it out at memfault.com  — it’s like having a detective in your device.

CI Metrics

CI Metrics

  • Show off your Pipeline
  • Track it all
    • Binary size, PR sizes vs time to review,  CI run times

  • Brag About It or draw attention
    • ​Put your metrics on display—literally. Make sure everyone can see them so you can fix issues before they blow up.
